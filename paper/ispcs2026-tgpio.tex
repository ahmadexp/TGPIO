\begin{abstract}
Clock synchronization protocols make time comparable across systems, but measurement and control applications also need a way to attach that time to physical signals. Precision Time Measurement (PTM) and Precision Time Protocol (PTP) can provide timing relationships for host, endpoint, and network clocks; Time-Aware GPIO (TGPIO) is important because it provides the complementary timed I/O boundary, capturing external edges and generating scheduled output transitions in the platform time domain. This paper presents a Linux driver that exposes firmware-undeclared Intel TGPIO hardware through the standard PTP hardware clock interface. The driver maps known MMIO resources, configures TGPIO blocks for external timestamp input or periodic output, and integrates the hardware's Always Running Timer (ART) timebase with Linux timekeeping. It supports realtime, raw ART, and adjustable PHC clock modes, allowing synchronized platform time to be observed, disciplined, and exercised at external pins with existing PTP user-space tools. A motivating application is connecting a GPS receiver's pulse-per-second output directly to a TGPIO input so that GPS-derived time can be made available to the host with minimal additional timing hardware, allowing the operating system to participate in a Time-Sensitive Networking (TSN) timing domain. We describe the driver architecture, pin-function model, timestamp conversion path, and output scheduling mechanism, along with practical issues encountered during bring-up, including firmware discovery gaps, ART/TSC ratio handling, input edge defaults, and output semantics established through register-level hardware experiments: single-shot output compares that never fire, an unreadable output level flop that holds toggle history across disables, and glitch-free in-flight rewrites of the periodic interval and compare registers. Building on those semantics, the driver disciplines a free-running hardware output without waveform restarts and tracks output polarity in software. Measured with a logic analyzer against the atomic-clock-referenced PPS of an OCP TimeCard, the disciplined TGPIO 1~PPS output achieves a mean offset of $+9$~ns with a standard deviation of 64~ns, statistically comparable to the TimeCard's own PPS output on the same instrument. The work demonstrates that TGPIO can turn an otherwise internal PTM/PTP-aware timing domain into usable physical I/O for synchronization, measurement, and control.
\end{abstract}

\section{Introduction}

Precision timing systems often need more than a synchronized network clock. They also need physical I/O that can bind events in the external world to a shared timebase. Examples include timestamping a pulse-per-second signal, measuring a trigger edge from an instrument, or generating a scheduled output transition for a device under test. Linux already provides a mature PTP hardware clock (PHC) abstraction for these ancillary timing functions, including external timestamp input and periodic output generation \cite{linux-ptp-docs}.

A concrete example is a small system that needs accurate time from a GPS receiver but does not have a network interface card PHC, FPGA, or dedicated timing card. Many GPS modules expose a PPS output with low timing uncertainty relative to the receiver's time solution. If that PPS edge can be timestamped directly by TGPIO and exposed as a PHC external timestamp event, linuxptp tooling can discipline the TGPIO PHC and then transfer that time to the system clock. This turns TGPIO into a minimal-hardware path from an external time reference into Linux timekeeping. It can also let the operating system participate in the timing domain of a Time-Sensitive Networking (TSN) system, where synchronized time is a foundation for time-sensitive applications and scheduling \cite{ieee8021as}.

Some commodity platforms contain timed I/O resources that are not exposed to the operating system through normal firmware enumeration. In that case, standard platform-driver discovery fails even when the underlying MMIO register block is present and functional. The result is an awkward gap: the hardware can be useful for synchronization experiments and measurement, but applications have no standard way to reach it.

This paper describes an out-of-tree Linux driver that bridges that gap for Intel Time-Aware GPIO (TGPIO) blocks on a tested desktop platform. The driver maps known static MMIO resources, exposes each block as a PTP pin, and allows the block to be configured as an external timestamp input, a periodic output generator, or unused. It is designed around the Linux PHC interface rather than a custom userspace ABI, so existing tools such as \texttt{testptp} can configure pins, enable external timestamp events, and request periodic output. A companion Python library, PyPTM, wraps the same public PTP character-device interface for scripted experiments and combines it with CPU timing primitives such as \texttt{RDTSC}, \texttt{RDTSCP}, CPUID feature checks, and \texttt{TPAUSE} where supported \cite{pyptm-repo}.

The central design problem is not only register access. TGPIO capture and compare values are expressed in the platform ART domain, while the PTP clock interface normally presents timestamps in a PTP clock domain. The driver therefore defaults to an ART-backed adjustable PHC and also provides explicit realtime and raw ART-derived modes for compatibility and diagnostics. The implementation also handles practical bring-up issues that are easy to miss in a purely architectural description, including default input edge selection, deterministic output startup, and output polarity.

The main contributions of this work are:
\begin{itemize}
\item a PTP-clock abstraction for static TGPIO MMIO blocks that firmware does not enumerate;
\item per-block mode selection for external timestamp input, periodic output, or disabled operation;
\item a timestamp path that supports both realtime-converted hardware captures and raw ART-derived timestamps;
\item a default ART-backed adjustable PHC mode suitable for external-reference tools such as \texttt{ts2phc};
\item experimentally established TGPIO output semantics (toggle-only level state, inert single-shot compares, safe hot register rewrites) and a discipline design built on them: grid-aligned re-arms, in-flight interval refresh, sub-100~ns phase nudges, and software-tracked output polarity;
\item a one-shot external calibration path that cancels constant clock-comparison and pipeline delays, bringing a disciplined 1~PPS output within 100~ns of an atomic reference; and
\item a reproducible bring-up workflow using standard Linux kernel and PTP user APIs, including GPS PPS input as a target application.
\end{itemize}

\section{Background}

\subsection{Linux PTP Hardware Clocks}

Linux exposes hardware-assisted precision timing through the PHC subsystem. A PHC driver registers a \texttt{struct ptp\_clock\_info}, implements operations such as \texttt{gettime64()} and \texttt{enable()}, and advertises ancillary features including external timestamp channels and periodic outputs. Userspace configures those features through the PTP character device interface. For this work, the key pin functions are \texttt{PTP\_PF\_EXTTS}, which emits timestamped edge events, and \texttt{PTP\_PF\_PEROUT}, which requests a periodic output signal \cite{linux-ptp-docs,linux-ptp-header}.

The PHC interface is a good boundary for TGPIO because it avoids inventing a new control plane for a timing device. It also allows the driver to interoperate with existing test and measurement programs. In the implementation described here, the registered clock reports the name Intel TGPIO; the PTP clock's time-read operation returns adjusted PHC time by default or realtime in explicit realtime mode; and pin assignment controls which physical TGPIO block is used for external timestamp input or periodic output.

\subsection{ART, TSC, and Cross Timestamps}

Modern Intel platforms include timing relationships among the Time Stamp Counter (TSC), the ART, and system time. CPUID leaf \texttt{0x15} can report a crystal clock frequency and a TSC-to-crystal ratio on supported processors \cite{intel-sdm}. Linux timekeeping also exposes mechanisms for relating hardware counter values to system time, including cross-timestamp support for device counters \cite{linux-timekeeping-docs,linux-crosststamp-rfc}.

The TGPIO hardware used by this driver captures and compares ART-domain counter values. For external timestamp input, the driver can either convert captured ART cycles into realtime nanoseconds or expose raw ART-derived nanoseconds. For periodic output, the driver converts a requested realtime edge time into an ART compare value before programming the TGPIO compare registers.

\subsection{Firmware Enumeration Gap}

Upstream and public discussion around Intel timed I/O shows that TGPIO-like facilities have existed in Linux development contexts \cite{linux-pps-gen-tio,lkml-tgpio-pmc,lkml-tgpio-io}. However, the platform studied here did not enumerate the relevant ACPI device for the operating system. The driver therefore treats the MMIO base addresses as platform-specific module parameters. This is less general than firmware discovery, but it is a useful technique when the hardware exists and the address map has been confirmed.

\section{Driver Design}

\subsection{Static Blocks and Module Parameters}

The driver supports two static TGPIO blocks. Each block has a configurable MMIO base address, a common MMIO span, and an independent mode. The default addresses are \texttt{0xFE001210} and \texttt{0xFE001310}, with a default register span of \texttt{0x38} bytes. Each block can be loaded in \texttt{input}, \texttt{output}, or \texttt{off} mode. The default configuration enables both blocks as rising-edge external timestamp inputs.

\subsection{PTP Pin Model}

At load time, the driver maps all enabled blocks and registers a single PTP clock. Each block becomes a PTP pin descriptor named according to its physical block index and selected mode. The PTP pin verification path accepts external timestamp requests only on blocks loaded in input mode and periodic-output requests only on blocks loaded in output mode. This prevents userspace from assigning a pin function that the selected hardware mode cannot support.

Channel mapping follows the PTP pin assignment model. A physical TGPIO block may be assigned to an external timestamp or periodic-output channel through \texttt{testptp -L}. If userspace does not create an explicit assignment, the driver falls back to a direct channel-to-block mapping where possible. This keeps the common two-block setup simple while still exercising the standard PTP pin API.

\subsection{Userspace Control}

The primary control surface is the standard PTP character device, not a private driver-specific ABI. This allows both existing command-line tools and custom measurement software to use the same kernel interface. During bring-up, \texttt{testptp} provided a useful reference for pin assignment, external timestamp enablement, and periodic-output requests. PyPTM provides the same operations as a Python API and command-line tool, making it easier to combine TGPIO configuration, event capture, and CPU-local timing measurements in one experiment script.

One compatibility detail first surfaced during userspace testing: the legacy \texttt{PTP\_EXTTS\_REQUEST} ioctl succeeded while \texttt{PTP\_EXTTS\_REQUEST2} returned \texttt{EOPNOTSUPP}, so PyPTM attempts the newer ioctl and falls back to the legacy request. The root cause was later identified in the driver: recent kernels validate the strict ioctl's flags against the \texttt{supported\_extts\_flags} field of \texttt{ptp\_clock\_info} before the driver sees the request, and a driver that leaves the field zero silently rejects every strict request. Advertising the supported edge and enable flags fixed \texttt{ts2phc} -- which uses only the strict ioctl -- and is worth checking in any PHC driver that predates the field.

\subsection{External Timestamp Input}

For input mode, enabling a PTP external timestamp request configures the TGPIO direction bit for input, applies an edge selection, clears the cached event count, and starts a delayed work item. The poll worker reads the capture timestamp and event counter from the block. The hardware register sequence matters: reading the low capture timestamp register latches the associated timestamp and event-count values, after which the driver reads the high and low count registers and the high timestamp register.

When the event count changes, the driver emits a PTP external timestamp event through the PHC event path. The event timestamp is chosen according to the timestamp mode. For debugging, a module option can bypass the captured hardware timestamp and emit the current poll time instead. Captures also feed several ancillary consumers: per-input statistics in the status file (the phase of each capture against the PTP clock's whole-second grid -- live discipline quality -- and min/max/mean spacing between captures), a standard RFC~2783 PPS source per input block so NTP-family tools can consume TGPIO captures without PTP awareness, and a time-to-digital converter mode that pairs start and stop captures across the two blocks into hardware-timestamped durations with one-ART-cycle resolution. The PHC additionally implements precise cross-timestamps (\texttt{PTP\_SYS\_OFFSET\_PRECISE}) with an exact device/realtime pairing, which \texttt{phc2sys} and chrony adopt automatically.

\subsection{Periodic Output}

For output mode, a PTP periodic-output request supplies a start time and period in the PTP clock's time domain. The driver rejects unsupported perout flags, validates the period, computes a half-period, and requires that Linux timekeeping expose an ART base clock. It then converts the requested start time into an ART compare value and the requested half-period into an ART interval value before programming the hardware.

Steady-state output uses the TGPIO hardware periodic feature: the comparator register holds the first active edge, the periodic interval register holds the half-period in ART cycles, and the periodic-mode bit lets hardware generate subsequent toggle transitions without software reprogramming each edge. Register-level experiments established that single-shot output compares never fire on this hardware and that the output level flop is write-only toggle history: it holds the level of the last generated transition, survives disable, and cannot be preloaded or read back. The driver therefore mirrors the flop in software. It assumes the power-on low state at load, folds every generated toggle into the mirror using the live comparator readback (the comparator advances by exactly one interval per edge), drains the flop low before stops and unloads so the assumption survives module reloads, and selects the first compare slot from the mirror so the rising edges of the waveform always land on the requested period grid. A guard pushes a late first compare forward by whole periods, since enabling with a compare already in the past fires the first toggle immediately and would invert the waveform. If external interference ever falsifies the mirror, a debugfs control flips it and shifts a running waveform by half a period without a glitch. To avoid programming an edge too close to the current time, the driver enforces a lead time before arming the first active transition. Asymmetric duty cycles whose halves are both at least 50~ms also run on the toggle engine: the interval register alternates between the on-time and off-time cycle counts, rewritten by a per-edge service inside each half, so every edge remains hardware-timed (a 25\% duty request measured 249.998~ms/749.995~ms on the analyzer); the earlier high-resolution timer re-arm path remains only for narrower pulses. The same engine provides one-shot pulses -- a rising and falling edge pair at hardware precision with a scheduled stop in the following low half. A related negative result is worth recording: with a 50\% duty output, edge timestamps alone cannot distinguish rising from falling edges, so output polarity is fundamentally unobservable from an external reference on the second block; it becomes observable (and self-correcting) only in a loopback configuration where the input captures the output's own rising edges.

Quantization is reported, not hidden. Converting a request onto integer ART cycles rounds both the half period and the first edge, and every arm therefore writes one unconditional \texttt{output quantization} line to the kernel log: the requested and actual period, the half-period in ART cycles, the period rounding residual, and the first-edge rounding measured by reading the programmed compare value back through the clock conversion. At the 38.4~MHz ART rate one cycle is about 26~ns, so both residuals are bounded by half a cycle; the logged values double as a self-check that the conversion path rounds to nearest. The debugfs status file exposes the same quantities continuously, together with the live phase error of the pending edge and the tracked output level.

\section{Clock-Domain Handling}

The driver supports three PTP clock modes. The default mode exposes an ART-backed adjustable PHC with \texttt{settime64()}, \texttt{adjtime()}, and \texttt{adjfine()} support for tools that discipline a PHC from external timestamps, such as \texttt{ts2phc}. An \texttt{art} mode reuses the same anchor-plus-rate clock model but anchors it to \texttt{CLOCK\_MONOTONIC\_RAW} and rejects every adjustment (\texttt{EOPNOTSUPP}, zero \texttt{max\_adj}), providing an undisciplined hardware-paced timebase; its rate is calibrated against the raw clock at load and refined once over a longer baseline. A GPS PPS signal connected directly to TGPIO can therefore become the external reference for the TGPIO PHC, after which other Linux time tools can make that disciplined time available to the system. The ART mode's rate accuracy depends on its calibration source: calibrating against \texttt{CLOCK\_MONOTONIC\_RAW} can only recover the nominal ratio, because that clock derives from the same crystal as ART, so the output retains the crystal's ppm-scale error against true seconds; an alternative realtime calibration tracks the kernel timekeeper's rate with a sliding-window estimator refined every 30 seconds -- as accurate as the system clock discipline while the phase still free-runs -- and a runtime trim parameter accepts an operator calibration measured on an external instrument (the observed time-error slope, written in ns/s with its sign). An explicit realtime mode is also available; in that mode the PTP clock's \texttt{gettime64()} operation returns \texttt{CLOCK\_REALTIME}, making external timestamp events and periodic-output requests line up with ordinary Linux realtime values, and the PTP adjustment operations steer the system clock itself (\texttt{settime64} through \texttt{do\_settimeofday64}; offset and frequency through \texttt{do\_adjtimex}, resolved at load since the kernel does not export it to modules). Because the system clock is also steered behind the driver's back (chrony, \texttt{adjtimex}), realtime-mode hardware outputs are re-synchronized to the realtime grid once per second using the same glitch-free nudge machinery as PHC mode, with the phase read back exactly through the timekeeper's realtime-to-ART conversion. A reference pulse captured by a TGPIO input can then discipline the operating-system clock directly from hardware timestamps; validated with a chrony PHC-refclock configuration, the system clock locked to the atomic PPS with a $-218$~ns offset and 11~ns per-sample standard deviation. In a TSN deployment, the default PHC mode gives the OS a local disciplined timebase for host-side timestamping, scheduling, and control logic, while network transmission still depends on the capabilities of the attached network interface and TSN stack. Internally, the TGPIO registers use ART-domain counter values for both capture and compare in both modes.

For input events in realtime mode, the driver constructs a device cross-timestamp context with the captured ART cycle count and the Linux ART clocksource identifier. It then calls the kernel cross-timestamp interface and uses the returned realtime value as the PTP event timestamp. The driver snapshots system time before reading the capture registers and reuses that history as a hint for the next conversion. In PHC mode, captured ART cycles are converted into the adjusted PHC time domain by applying the current PHC offset and frequency scale.

The alternate raw ART timestamp mode converts ART cycles to nanoseconds using the configured or auto-detected ART frequency. This mode is useful for debugging the hardware counter scale but intentionally does not claim realtime epoch alignment. CPUID leaf \texttt{0x15} is used to detect the ART frequency when available and to report the TSC/ART ratio for diagnostics. The TSC/ART ratio is not applied to TGPIO capture values because those values are already treated as ART-domain cycles.

For output events, clock-to-ART conversion is required because the userspace request is expressed in the PTP clock domain while the hardware compare and interval registers expect ART cycles. In realtime mode, absolute edge-time conversion uses Linux timekeeping's realtime-to-ART relationship. In PHC mode it uses the driver's stable ART base rate plus the adjusted PHC offset and frequency scale. Hardware periodic output uses the same clock-domain conversion for the periodic interval register. If the required timekeeping relationship or base rate is unavailable, the driver fails the request rather than scheduling an edge in the wrong domain.

A running hardware output must also follow PHC discipline. The PHC base rate is calibrated from the kernel timekeeper at load rather than taken from the nominal CPUID crystal frequency, which on the tested platform differs from the effective rate by tens of parts per million. When a tool steps the PHC, the driver re-arms the output with the first edge advanced along the originally requested period grid (start plus an integer number of periods), so a step never shifts the waveform phase. When a tool adjusts only frequency, the driver rewrites the periodic interval register in flight -- experiments confirmed such writes latch cleanly for subsequent periods -- and compares the pending edge, read live from the comparator, against the period grid; residual errors above 50~ns are corrected by rewriting the comparator, which moves only the pending edge and produces no glitch. Finally, a runtime-writable calibration offset shifts every programmed edge by a constant, letting a one-shot external measurement cancel systematic delays such as the asymmetric PCIe read latency in PHC-to-PHC comparison; the phase readback applies the inverse, so the nudge machinery converges a running output onto the calibrated position within a few update cycles.

\section{Bring-Up Lessons}

Several implementation details were discovered through bring-up rather than from the initial driver outline. First, the input default was changed to rising edge. Both-edge capture can be useful for debugging, but a rising-edge default better matches common PPS and trigger workflows and avoids accidentally doubling the apparent event rate.

Second, realtime timestamping, raw ART timestamping, and hardware periodic output need different dependency checks. Raw ART conversion and hardware periodic intervals require a frequency value, either supplied by the user or auto-detected through CPUID. Realtime timestamp conversion instead relies on Linux timekeeping's ART base-clock relationship. As a result, the driver reports CPUID frequency and ratio values for diagnostics even when a particular realtime-only input configuration does not mathematically require the manual ART frequency scale.

Third, the output hardware semantics differed substantially from the initial mental model, and only direct register experiments observed with a logic analyzer settled them. Single-shot (non-periodic) output compares never fire, so an initially implemented ``preconditioning edge'' was a silent no-op. The edge-polarity field has no effect on outputs; in periodic mode the pin simply toggles at each compare match. The output level itself lives in a write-only flop that retains the last generated level across disables, is re-driven on enable, and cannot be preloaded through the edge-polarity field, cleared through direction round-trips, or observed through any register. Releasing the pin by switching to input mode floats it toward mid-rail and is unusable as a readback. Each wrong hypothesis was falsifiable within minutes because the analyzer captures were scripted; this loop of poke, capture, and decode was the single most productive bring-up tool.

Fourth, in-flight register rewrites turned out to be safe and useful. A periodic-interval rewrite on a running block latches for the following periods, and a comparator rewrite moves only the pending edge. One caveat applies: a comparator rewrite can register a phantom event that silently flips the level-mirror parity, so the driver pauses the periodic-mode bit (output enable stays set, so the line holds its level) around such rewrites. An earlier belief that such writes corrupt the waveform originated from testing an older driver revision and dissolved under controlled experiments. These two properties enable frequency and phase discipline of a free-running output without ever restarting the waveform.

\section{Evaluation}

The tested platform is an ASUS ProArt Z890-CREATOR WIFI motherboard with BIOS version 3202 running Ubuntu 26.04 LTS with Linux \texttt{7.0.0-27-generic}. The default static TGPIO addresses and two-block setup were validated on that system. The timing reference is an OCP TimeCard whose PHC (\texttt{ptp\_ocp}) is fed by its onboard atomic clock. A Saleae Logic Pro 16 logic analyzer, sampling four channels at 100~MS/s (10~ns resolution), simultaneously observes the atomic-clock 1~PPS, the TimeCard PHC's PPS output, and both TGPIO periodic outputs. The analyzer's own crystal error cancels because all results are channel-to-channel differences against the atomic PPS. \texttt{phc2sys} disciplines the TGPIO PHC from the TimeCard PHC at eight updates per second.

Table~\ref{tab:eval} summarizes the output results. Undisciplined, the TGPIO output tracks the system clock with the expected free-running error. Once \texttt{phc2sys} locks the TGPIO PHC to the TimeCard, the interval-refresh path removes the rate error entirely: over two-minute captures the measured mean period is identical to the atomic reference at the resolution of the instrument, with zero waveform restarts and the driver-reported phase error below 100~ns. The residual $+2.38$~$\mu$s mean offset before calibration is constant (drift below 1~ns/s) and is dominated by asymmetry in the PHC-to-PHC comparison path: reading the TimeCard PHC across PCIe costs several microseconds round trip while the TGPIO PHC read is a fast timekeeping conversion, and \texttt{phc2sys} necessarily assumes symmetric readings. Writing the externally measured constant into the calibration parameter moves the output to $+9 \pm 64$~ns against the atomic PPS -- statistically comparable to the TimeCard's own PPS output measured on the same instrument ($-10 \pm 54$~ns). The remaining spread is consistent with the 26~ns ART quantization step plus servo wander of the software PHC. Reaching this floor also required pinning the platform: holding \texttt{/dev/cpu\_dma\_latency} at zero and selecting the performance cpufreq governor stabilized the PCIe read latency that the PHC comparison depends on, and the comparison constant must be recalibrated if that latency regime changes.

Polarity determinism was validated separately: across five consecutive module reloads, deliberate 300~ms PHC steps, and \texttt{phc2sys} restarts, both outputs came up with rising edges on the requested period grid every time, and a deliberately falsified level mirror was corrected through the debugfs control while running, snapping a $-500$~ms inverted waveform onto the grid without a glitch. The input-capture, GPS PPS, and timestamp-mode comparisons from the original evaluation plan remain future measurements; the workflow (scripted analyzer captures cross-checked against driver activity logs and the debugfs status file) is directly reusable for them.

The hardware-domain loop was subsequently closed: the reference PPS was wired into a TGPIO input and \texttt{ts2phc} disciplined the TGPIO PHC from the block's own ART-domain external timestamps, with the second block generating the output. Because capture and generation share the ART timebase, the PCIe comparison path drops out of the loop entirely: the uncalibrated systematic offset collapses from $2.4$~$\mu$s to about $+50$~ns (the residual capture-versus-generate pipeline difference, roughly two ART cycles), and no platform latency pinning is required. Measured output alignment in this mode was $-52 \pm 122$~ns against the reference PPS distribution, whose own wire-to-wire noise floor on the instrument was 60--80~ns; both discipline paths therefore land within about $\pm 100$~ns of the atomic reference, with the input-capture path preferable operationally because its systematic error needs no external calibration. This same configuration is the complete GPS use case with the atomic PPS standing in for a GPS receiver.

The output path was additionally cross-validated on an independent instrument. A Calnex Sentinel measured 1~PPS time error simultaneously on both TGPIO outputs and on the atomic reference, with per-second differencing against the reference channel cancelling the instrument's free-running internal timebase. With both blocks generating disciplined 1~PPS outputs (phc2sys path), a 15-minute run measured $-42 \pm 103$~ns and a subsequent 7-minute run $-66 \pm 83$~ns against the atomic PPS, with residual drift below $0.25$~ns/s; each channel's absolute spread stayed under 80~ns, and the differencing adds the two channels' trigger noise in quadrature. Repeated windows show the PCIe-path systematic wandering on a $\pm 60$~ns scale over tens of minutes, which bounds how finely the one-shot delay calibration is worth trimming. The alternate clock modes were characterized in the same common-view setup (Fig.~\ref{fig:modes}). ART mode with the default same-crystal calibration drifts at the crystal error ($-41$~$\mu$s/s measured); with realtime calibration the sliding-window estimator holds the residual rate to tens of ppb, lagging crystal temperature transients by a few minutes, and the operator trim can remove the remainder after one instrument measurement. Realtime-mode outputs, re-synchronized once per second, track the system clock faithfully -- which makes the quality of the system clock discipline the dominant term: an aggressive 8~Hz linreg \texttt{phc2sys} dithers the clock frequency and widens the output band to $\pm 250$~ns, while a 1~Hz PI servo against the PTM master holds the pin within about $\pm 150$~ns of the atomic-aligned grid.

Finally, enabling PCIe Precision Time Measurement on the master card (a gateware update exposing hardware cross-timestamps through \texttt{PTP\_SYS\_OFFSET\_PRECISE}) removed the software-bracketed read asymmetry from the phc2sys comparison entirely: both legs of the comparison then rest on hardware-paired timestamps, the microsecond-scale delay calibration becomes unnecessary, and a 22-minute common-view run with zero calibration showed smoothed output phase wander of about 10~ns RMS against the shared atomic reference with drift below $10^{-11}$ (Fig.~\ref{fig:ptmrun}). Tightening the driver's phase-nudge dead-band from its 200~ns default to 30~ns (a runtime parameter; each nudge is one of the verified glitch-free in-flight compare rewrites) and deepening the phc2sys per-sample read filtering reduced each output's absolute spread on the instrument from 78~ns to 47~ns -- below the 69~ns spread of the atomic reference channel itself on the same instrument -- with residual drift of $2$~ps/s.

\section{Limitations and Future Work}

The current implementation is intentionally platform-specific. It requires known MMIO addresses and should only be loaded on systems where those addresses have been confirmed. A production-quality upstream driver would need robust platform discovery, ownership checks, and maintainable quirks rather than hard-coded defaults.

The external timestamp path is polling-based. Polling keeps the driver compact and was sufficient for bring-up, but it creates a latency and event-loss tradeoff controlled by \texttt{poll\_ms}. An interrupt-backed capture path would be preferable for high-rate event streams or applications that need lower software latency.

The default PHC mode adjusts a software clock model layered on ART rather than disciplining the ART oscillator itself. This is useful for tools that require PHC adjustment semantics, but it differs from a network interface card PHC that owns and disciplines an independent hardware clock.

Output polarity ultimately rests on one bit of external truth. The hardware level flop cannot be read, so the software mirror is seeded with the power-on assumption and preserved by draining the flop low across stops and reloads; anything that toggles the pin outside the driver (or a cold state the driver never saw) requires a one-shot external observation to correct through the debugfs flip control. Relatedly, tools holding the PTP character device open, such as \texttt{phc2sys}, must be restarted after a module reload because the clock device is recreated.

Future work includes validating the driver across additional platforms, replacing static addressing with a discovery or quirk mechanism, adding interrupt support where available, and evaluating whether a subset of the design is suitable for upstream Linux review.

\section{Conclusion}

This work shows that firmware-undeclared timed I/O hardware can be made useful through the standard Linux PTP interface. By mapping known TGPIO MMIO resources and presenting each block as a PTP external timestamp input or periodic output pin, the driver lets existing tools control hardware that would otherwise be inaccessible to userspace. The GPS PPS use case illustrates the practical value: a direct TGPIO input can provide a compact path from an external time reference into host timekeeping, allowing the OS to become a time-aware participant in a TSN system with minimal hardware.

The most important engineering issues are clock-domain correctness and deterministic output behavior. TGPIO capture and compare values are ART-domain cycles, while PTP userspace operates naturally in realtime. The driver addresses this with explicit timestamp modes, Linux timekeeping cross-timestamp conversion, realtime-to-ART output scheduling, and a software mirror of the write-only hardware level state that keeps output polarity deterministic across arms, steps, and reloads. The measured results show that with in-flight interval refresh, live comparator phase nudges, and a one-shot delay calibration, a disciplined TGPIO output holds an external atomic PPS reference within 100~ns -- comparable to a dedicated timing card's own output -- using only hidden platform hardware and standard PTP tooling. The resulting design provides a practical bridge for measurement, synchronization, and control experiments on platforms with hidden timed I/O resources.
